# Rental AI Chatbot

Прототип AI-агента для автоматизации общения с потенциальными арендаторами по объявлениям о долгосрочной аренде. Агент ведёт диалог по сценарию, отвечает по базе знаний объявления, мягко работает с возражениями и подводит пользователя к записи на просмотр.

**Живое демо:** [domclick-agent.vercel.app](https://domclick-agent.vercel.app)

## Контекст

Проект собран как demo-ready MVP AI-ассистента для переписки в интерфейсе, знакомом пользователю Avito. Цель не в том, чтобы показать “ещё один чат на LLM”, а в том, чтобы показать управляемую продуктовую логику: state machine диалога, structured knowledge base, аналитику и админку для просмотра качества лидов.

---

## Что умеет

- Отвечает на вопросы строго по базе знаний объявления — без галлюцинаций
- Ведёт диалог через 7 стадий: от знакомства до записи на просмотр
- Распознаёт намерения, интерес и возражения арендатора
- Поддерживает 5 объявлений с разными агентами и контекстом
- Адаптивный интерфейс под Авито — работает на мобиле и десктопе
- Панель управления с метриками, воронкой и AI-саммари каждого диалога

## Архитектура: путь одного сообщения

```
Пользователь отправляет сообщение
        │
        ▼
POST /api/chat  {message, sessionId, listingId}
        │
        ├─► Получить / создать conversation в Supabase
        │
        ├─► Загрузить историю (последние 10 сообщений)
        │
        ├─► [LLM #1] Extraction (temp 0.1)
        │   Вход:  сообщение + история
        │   Выход: intent, interest_level, sentiment,
        │          has_objection, readiness_to_view
        │
        ├─► Вычислить следующую стадию (state machine)
        │
        ├─► Построить стратегию ответа
        │
        ├─► [LLM #2] Generation (temp 0.7)
        │   Вход:  стадия + стратегия + KB + история + тон
        │   Выход: финальная реплика агента
        │
        ├─► Сохранить сообщение и обновить conversation
        │
        └─► {reply, stage, signals, responseTime}
```

### Почему два LLM-вызова

Один вызов хуже справляется: модель одновременно анализирует и отвечает, что повышает риск галлюцинаций и смешивает логику с генерацией.

Два раздельных вызова дают:
- **Контроль** — сигналы extraction хранятся в БД и используются в аналитике
- **Надёжность** — generation работает с уже верифицированными фактами
- **Дебаггабельность** — каждый слой можно улучшать независимо

## State machine диалога

```
greeting ──► qualification ──► info_exchange ──► soft_nudge ──► appointment ──► closed
                                      │                                ▲
                                      └──► objection_handling ─────────┘
```

| Стадия | Триггер перехода |
|--------|----------------|
| `greeting → qualification` | Первое сообщение получено |
| `qualification → info_exchange` | Заданы вопросы по квартире |
| `info_exchange → soft_nudge` | Высокий `interest_level` или `readiness_to_view` |
| `любая → objection_handling` | `has_objection = true` |
| `любая → appointment` | `intent = appointment_request` |
| `любая → closed` | `intent = farewell` |

## База знаний: structured JSON вместо RAG

Для пилота с фиксированным числом объявлений RAG избыточен. Structured JSON в `data/listings.ts` даёт:

- **Нулевые галлюцинации** — агент видит только факты объявления, ничего лишнего
- **Быстрый старт** — нет embeddings, векторной БД, retrieval-пайплайна
- **Простое масштабирование** — добавить объявление = добавить объект в массив
- **Контроль** — легко проверить что именно знает агент о каждой квартире

При масштабировании на сотни объявлений эту схему можно заменить на RAG с pgvector (уже встроен в Supabase).

## Схема БД

```sql
conversations          -- один диалог на сессию арендатора
  session_id           -- уникальный ID сессии
  listing_slug         -- к какому объявлению относится
  stage                -- текущая стадия диалога
  signals              -- последние extraction-сигналы (JSON)
  sentiment            -- общий тон разговора
  appointment_booked   -- была ли записана на просмотр

messages               -- все сообщения диалога
  conversation_id      -- FK на conversations
  role                 -- user | assistant
  content              -- текст сообщения
  intent               -- extraction-сигналы этого сообщения
  response_time_ms     -- время ответа агента

settings               -- глобальные настройки (single-row)
  tone_preset          -- friendly | formal | warm
```

## Ключевые архитектурные решения

**1. KB в коде, а не в БД-CMS** — минимизирует время на MVP, делает ответы предсказуемыми и легко проверяемыми.

**2. Multi-listing через query param** — один чатовый интерфейс обслуживает 5 объявлений без дублирования кода. `/chat?listing=presnya-3room` загружает контекст 3-комн. на Пресне, агента Дмитрия и его KB.

**3. AI summary on demand** — саммари генерируется только по клику в админке, не сохраняется в БД. Упрощает схему и не тратит токены на неиспользуемые диалоги.

**4. Антигаллюцинационные правила в промпте** — явные инструкции не утверждать договорённости, которых не было, и говорить "уточню у хозяйки" если факта нет в KB.

**5. Авито-интерфейс без имитации несуществующего** — UI намеренно выглядит как Авито-мессенджер, но не притворяется функциями которых нет: авторизация, push-уведомления, реальные фото — всё явно является демо-данными.

## Локальный запуск

```bash
# Клонировать репозиторий
git clone <repo-url>
cd domclick-agent

# Установить зависимости
npm install

# Заполнить переменные окружения
cp .env.example .env.local
# Заполнить GROQ_API_KEY, NEXT_PUBLIC_SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY

# Применить схему БД
# Открыть Supabase SQL Editor и выполнить supabase/schema.sql

# Запустить
npm run dev
```

Открыть [localhost:3000](http://localhost:3000)

Панель управления: [localhost:3000/admin](http://localhost:3000/admin) — PIN `1234`

## Что можно развить

- **RAG вместо structured KB** — при масштабировании на 100+ объявлений подключить pgvector и embeddings
- **Авторизация арендодателя** — Supabase Auth вместо PIN
- **Веб-хук из реального Авито** — подключить к настоящему чату через API Авито
- **A/B тесты тона** — сравнивать конверсию при разных tone preset
- **Мультиязычность** — промпты уже изолированы в `lib/prompts.ts`, легко добавить EN
