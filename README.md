# AI-агент для аренды квартиры

Живое демо: [domclick-agent.vercel.app](https://domclick-agent.vercel.app)  
Репозиторий: [github.com/iDaveX/rental-ai-chatbot](https://github.com/iDaveX/rental-ai-chatbot)

Это тестовое задание для Domclick на роль `VibeCoder`.

Суть проекта простая: я собрал AI-агента, который ведёт переписку с потенциальным арендатором по объявлению о долгосрочной аренде. Агент не просто отвечает на вопросы, а держит контекст квартиры, говорит в заданном тоне, опирается на базу знаний и мягко подводит человека к записи на просмотр.

Снаружи это выглядит как знакомый интерфейс объявлений и чата. Внутри это уже не “ещё один LLM-чат”, а управляемый продукт с логикой диалога, аналитикой и админкой.

## Что здесь можно посмотреть

- витрину объявлений в стилистике Авито;
- переписку арендатора с AI-агентом по конкретной квартире;
- разные объекты с разным контекстом и разными персонажами агентов;
- админку с метриками, воронкой, саммари диалогов и настройкой тона;
- ручной сценарий проверки “как будто это уже пилот”.

## Что именно умеет продукт

- отвечает на вопросы по объявлению в формате живого диалога;
- учитывает конкретную квартиру, а не отвечает абстрактно;
- использует встроенную базу знаний по каждому объекту;
- умеет вести разговор в разном тоне;
- определяет стадию диалога и не просто реагирует, а ведёт к просмотру;
- показывает администратору обращения, скорость ответа, эмоциональный тон и записи на просмотр;
- формирует краткое AI-саммари по лиду прямо в админке;
- скрывает подозрительные или токсичные сообщения из управленческой аналитики.

## Как быстро проверить проект

Если вы смотрите этот репозиторий как тестовое задание, самый короткий путь такой:

1. Откройте [domclick-agent.vercel.app](https://domclick-agent.vercel.app)
2. Выберите объявление `2-комн. квартира, 54 м², 7/12 эт.`
3. Напишите в чат, например:
   - `Можно с кошкой?`
   - `Когда можно посмотреть в субботу?`
   - `Мне подходит суббота в 12:00, запишите меня`
4. Затем откройте [domclick-agent.vercel.app/admin](https://domclick-agent.vercel.app/admin)
5. Введите PIN: `1234`
6. Нажмите `Обновить`
7. Посмотрите:
   - свежий диалог в ленте;
   - ключевые метрики сверху;
   - блок `Саммари`;
   - настройки активных агентов внизу.

Этого достаточно, чтобы понять весь продукт за пару минут.

## Как устроен продукт

### 1. Витрина объявлений

На главной странице находится каталог квартир. Для тестового это важно не только как UI, но и как способ показать, что агент работает не с одной захардкоженной карточкой, а с несколькими разными объектами.

У каждого объявления есть:

- цена;
- район и адрес;
- характеристики квартиры;
- свой агент;
- свой стиль общения;
- собственная база знаний.

### 2. Чат арендатора

После перехода в чат пользователь общается не с “безликой моделью”, а с агентом, который:

- знает только факты по конкретному объекту;
- не выдумывает того, чего нет в knowledge base;
- отвечает естественно и по-человечески;
- адаптирует подачу под выбранный tone;
- постепенно ведёт разговор к просмотру.

Внутри используется управляемая логика стадий диалога:

`greeting → qualification → info_exchange → soft_nudge → appointment → closed`

Отдельно обрабатывается сценарий возражений:

`objection_handling`

То есть агент не просто генерирует реплику “на глаз”, а двигается по понятной продуктовой логике.

### 3. Админка

Панель управления нужна не для красоты, а для реальной проверки пилота.

В ней можно увидеть:

- сколько пришло обращений;
- сколько сообщений отправили арендаторы;
- среднюю скорость ответа;
- сколько разговоров дошло до записи;
- текущую воронку;
- тональность по диалогам;
- раскрытый диалог целиком;
- AI-саммари по каждому лиду;
- настройку tone по каждому агенту.

Админка обновляется вручную по кнопке `Обновить`. Автообновление специально убрано, чтобы интерфейс был предсказуемым и не ломал демонстрацию.

## Архитектура в одном абзаце

Когда пользователь пишет сообщение, сервер:

1. находит нужный диалог или создаёт новый;
2. сохраняет входящее сообщение;
3. извлекает сигналы из текста арендатора;
4. определяет следующую стадию диалога;
5. строит стратегию ответа;
6. генерирует финальную реплику;
7. записывает результат в базу;
8. обновляет аналитику разговора.

Такой пайплайн нужен, чтобы продукт был управляемым: отдельно логика, отдельно генерация, отдельно аналитика.

## Почему база знаний сделана так

Для этого кейса я сознательно не строил тяжёлый RAG-контур.

Вместо этого у каждого объявления есть структурированная knowledge base в коде. Это даёт несколько практических плюсов:

- меньше риска галлюцинаций;
- проще проверять ответы;
- быстрее запускать MVP;
- легче показать управляемость решения;
- удобнее поддерживать небольшой пилот с несколькими объектами.

Если бы задача была про масштабирование на сотни или тысячи объявлений, следующим шагом был бы перенос знаний в RAG/pgvector-контур.

## Технологии

- `Next.js 14` для фронтенда и API
- `React 18`
- `Tailwind CSS`
- `Supabase / PostgreSQL` для хранения диалогов, сообщений и настроек
- `Anthropic Claude` как основной LLM
- `Groq` как резервный провайдер на случай недоступности основной модели
- `Vercel` для деплоя

app/
  page.tsx                  главная витрина объявлений

  admin/page.tsx            админка

  schema.sql                схема базы данных

## Локальный запуск

```bash
git clone https://github.com/iDaveX/rental-ai-chatbot.git
cd rental-ai-chatbot
npm install
cp .env.local.example .env.local
```

Заполните `.env.local`:

```env
ANTHROPIC_API_KEY=
GROQ_API_KEY=
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
```

Затем:

1. создайте проект в Supabase;
2. выполните SQL из `supabase/schema.sql`;
3. запустите проект:

```bash
npm run dev
```

После этого будут доступны:

- `http://localhost:3000`
- `http://localhost:3000/admin`

PIN для входа в админку в демо-режиме: `1234`

## Что важно понимать

Это не интеграция с реальным Авито и не production-сервис. Это демонстрационный, но рабочий продуктовый прототип, собранный так, чтобы по нему можно было честно оценить:

- качество AI-диалога;
- глубину работы с LLM;
- продуктовую логику;
- интерфейс;
- управляемость решения;
- готовность к реальному пилоту.
